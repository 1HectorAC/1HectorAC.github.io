ASP.NET Core Web API

An API for managing a library system. can do CRUD operations on Books, Users, Checkouts, Authors, Genre, Publishers, BookCopies, and BookGenres. Can also check-out/check-books. There is also login and Authorization restrictions for certain endpoints.

Get a better understanding of APIs and some key features around them.

5. Authentication

For many of the endpoints there are restricted access to only authenticated users. There is an endpoint to login where a user will pass in credentials (Email & Password) to the body and the endpoint will return a Json Web Token. This JWT can then be passed in the header of subsequent requests that require authentication to gain access. Passwords are also store as hashes using bcrypt since sensitive info should normally not be store as plain text. There are also a lot of setting that can be set around JWT but I used a minimal amount of restrictions so that I could quickly show it working. There also is no endpoint to register a user but rather an endpoint called addUser that can only be accessed by certain authenticated users. The idea behind this was that only an employee with the "employee" or "admin" role could make users. Here are the step to set things up:

  1. Add necessary packages:
  2. Added SECRET_KEY to .env file (for password hashing algorithm)
  3. Added authentication (with configurations) to DI Container. This will setup the use of JWT for Authentication and what will be checked.

Next I will go over how passwords are hashed. This part intertwines with the next part but I thought I would separated it for clarity. The Bcrypt library has a function to hash a password and to compare a password to its hashed value (for verification). The first is used when creating a user and the second is used for logging in. Example code:


//Hashing Password
Bcrypt.Net.Bcrypt.HashPassword(user.Password)
// Verifiy Password
BCrypt.Net.BCrypt.Verify(inputedPassword, storedHashedPassword)

After than I will go over how a JWT is made/returned in the Login endpoint. In the AuthController I first setup an IConfiguration variable to gain access to the environment variable SECRET_KEY. This is needed for creating signature for the token. Signature is just a string generating using an algorithm that takes the content and a secret key. The signature is used to make sure that the content wasn't tampered with given that it can't be generated without the secret key which only the server has. 


// Example of passing Iconfiguration in AuthController Contructor:
private readonly IConfiguration _configuration;

public AuthController( IConfiguration configuration)
{
    _configuration = configuration;
}

Below is the code in the Login endpoint that goes though the steps of generating/returning a JWT. Most of it is just setting things up like the roles, claims, algorithm used in signing, token configurations, etc.


// AuthController, in Login endpoint
List<Claim> claims = new()
{
    new Claim(ClaimTypes.NameIdentifier, user.Id.ToString()),
    new Claim(ClaimTypes.Email, user.Email)
};
var roles = user.Roles.Split(',');
foreach (var role in roles)
{
    claims.Add(new Claim(ClaimTypes.Role, role.Trim()));
}

// Setup Key and Credentials
var Secret_key = _configuration["SECRET_KEY"] ?? throw new InvalidOperationException("Secret key not found in configuration. Controller: AuthController, Method: Login");
var key = new SymmetricSecurityKey(System.Text.Encoding.UTF8.GetBytes(Secret_key));
var creds = new SigningCredentials(key, SecurityAlgorithms.HmacSha256);

// Create the Token
var token = new JwtSecurityToken(
    issuer: null,
    audience: null,
    claims: claims,
    expires: DateTime.Now.AddMinutes(30),
    signingCredentials: creds
);
var tokenString = new JwtSecurityTokenHandler().WriteToken(token);

return Ok(new { Token = tokenString });

Now that login capabilities were added the next part is adding restricted access to endpoints. This is done with the Authorize attribute (ex. "[Authorize]") at the top of a Controller or Action. This is added to many of the endpoints. Also note that some have specific Role access, as in only Users with a certain Role String can access them. This can be specified in the Authorize attribute (ex. "[Authorize(Roles="admin, employee")]").



6. Default Admin

While working on the project I realized that I needed to add a default admin user. This was because only authorized users with an "Admin" or "employee" role can add new users but if the database is empty at the start then nobody has access to it. I considered making an example script to make an admin user, I thought doing it in the application level made it more usable for the user. To do this I first added the environment variables "ADMIN_PASSOWORD" and "ADMIN_EMAIL" to the .env file. These are used to configure an initial admin user. Then, in the Program.cs file I added the configurations. The Configurations involve the following:

  1. Creates a container with scoped lifetime (this is needed to get access to DbContext which has a scoped lifetime)
  2. Check if any users with admin privilages exits using an instance of the dbContext
  3. If not exits, check If Enviroment Variables for password and email exits
  4. It yes, Create a user and save

// Program.cs, under build (var app = builder.Build();)
using (var scope = app.Services.CreateScope())
{
    var dbContext = scope.ServiceProvider.GetRequiredService<LibraryDbContext>();
    dbContext.Database.Migrate();

    if(!dbContext.Users.Any(i => i.Roles == "admin") )
    {
        Console.WriteLine("No admin user found. Creating default admin user...");
        var adminEmail = Environment.GetEnvironmentVariable("ADMIN_EMAIL");
        if(string.IsNullOrEmpty(adminEmail))
        {
            throw new InvalidOperationException("Admin email not set in environment variables.");
        }
        var adminPassword = Environment.GetEnvironmentVariable("ADMIN_PASSWORD");
        if(string.IsNullOrEmpty(adminPassword))
        {
            throw new InvalidOperationException("Admin password not set in environment variables.");
        }

        var adminUser = new LibrarySystem.Models.User
        {
            FirstName = "Admin",
            LastName = "Admin",
            Email = adminEmail,
            Password = BCrypt.Net.BCrypt.HashPassword(adminPassword),
            Roles = "admin"
        };

        dbContext.Users.Add(adminUser);
        dbContext.SaveChanges();
        Console.WriteLine("Admin user created with email: " + adminEmail);
    }
}


7. Caching

In this project, in memory caching was used to improve performance. A quick rundown on how it works is that certain requests that access the database will have the results stored in memory for a certain period of time. When subsequent request are made to the same endpoint the cache is checked first and retrieved from there instead of the database if it exits. In memory storage is faster than accessing the database so the result is a faster response. A few things to note is that there are different types of caching (in-memory, distributed) and different types of expiration policies (absolute, sliding, manual) that can be set when using caching. I chose to use in-memory caching and absolute expiration policy (expires after x amount of time) since they were the simplest option. Here are the steps to setup and use caching:

  1. Add In memory Service to DI container:
    2. 
builder.Services.AddMemoryCache();
  2. In controllers that will use caching, inject IMemoryCache into the contructor. Can also set a timeExpiration variable at the top 
    3. 
private readonly TimeSpan _cachExpiration = TimeSpan.FromMinutes(30);
  3. Example of saving cache value in memory:
    4. 
_cache.Set(cacheKey, value, _cacheExpiration);
  4. Example of removing value from cache:
    5. 
_cache.Remove(cacheKey);
  5. Example checking if book exists and getting
    6. 
if (!_cache.TryGetValue(cacheKey, out Book? book))
{
	// Book will be null here and will need to get from db
}
// book is retrieved from cache and book is set with its value

9. Final Thoughts

The big takeaways I had with this project was the security on Web APIs and the importance/usefulness of caching. In terms of web security I got a more in-depth understanding on how Json Web Tokens work and some of the security options around them. When it came to caching, I haven't used it before and using in-memory caching seems like a quick and easy way to improve performance with a database. A final important thing I learned was that focusing more on an in-depth understanding, playing around with features, and spending more time planning out a project played a big part in improving the speed I worked on this project. You really hit the ground running when you know what you want to do and how to do it ahead of time.
